CIDR:- classless inter domain routing 
1> IPV4 2> IPV6

These ranges are not routable on the public internet, which makes them safe for internal networking:

Range CIDR Notation IPs Available
10.0.0.0 10.0.0.0/8 ~16 million
172.16.0.0 172.16.0.0/12 ~1 million
192.168.0.0 192.168.0.0/16 ~65,000
✅ Why They're Always Used in VPCs
Avoid IP conflicts on the internet

Guaranteed uniqueness within your network

Supported by all networking devices and cloud providers

AWS VPCs let you choose any of these ranges, but 10.x.x.x is most common due to its flexibility (largest range)
Reserved IPs in Every Subnet (per CIDR block):
Let’s say your subnet is 10.0.0.0/24:


IP Address Purpose
10.0.0.0 Network address
10.0.0.1 Reserved for the VPC router
10.0.0.2 Reserved for DNS server
10.0.0.3 Reserved for future use
10.0.0.255 Network broadcast address (not used in AWS, but still reserved)
So, for a /24 subnet, although 256 IPs are available in theory, only 251 are usable.

NAT gateway ( Network address trans Gateway )
Nat gateway is to provide secure internet connection to private server in private subnet. (for example to download packages ,depedencies etc)
internet only acccessible from private server end . 
NAT gateway has static ip ( fixed elastic IP ). where private server reaches to this fixed elastic ip (where it leave request  & bring response with elastic fixed ip )
NAT gateway remember ip translation to serve server request properly.
NAT gateway present in public subnet 

Here’s the IP Translation Process (very simply):

Your EC2 (private subnet) has a private IP like 10.0.1.5.

It sends a request to the internet (example: apt-get update → connecting to archive.ubuntu.com).

At the NAT Gateway:

NAT sees the packet from 10.0.1.5.

NAT replaces the source IP (10.0.1.5) with its own Elastic IP (public IP) — e.g., 3.80.120.50.

It remembers this mapping (10.0.1.5 ↔ 3.80.120.50:port).

The packet goes to the internet — from NAT’s public IP (3.80.120.50).

Response from the internet (say from archive.ubuntu.com) comes back to NAT Gateway at 3.80.120.50.

At NAT Gateway:

NAT checks its mapping table.

It knows this reply is meant for 10.0.1.5.

It replaces the destination IP back to 10.0.1.5 and sends the packet into the VPC.

Your EC2 receives the response — it thinks it is directly talking to archive.ubuntu.com.

Route 53
route 53 is aws DNS service port 53.
3 concepts:- hosted zone, records ,routing policy 
1> hosted zone (public & private )
2>record 
    #A type record (hostname <->public ip ) simple one
    #Alias type record - (hostname <-> LB) routing via LB 
	#CName record   - (hostname <->domain <-> subdomain) A CNAME (Canonical Name) record is used in Amazon Route 53 to map one domain name (alias) to another domain name.
	for example www.example.com → app.example.net

3> routing policy 
   Simple routing , failover routing ,latency based routing , geolocation routing ,Weighted routing 

#Simple routing :- (basic routing ) sRoutes all traffic to a single endpoint .Single resource (e.g., one EC2 instance or one IP). for example :- www.example.com → 192.0.2.1
#Failover routing (high avaibility) :- Routes only primary source (region) . On failure switch to secondary source (region) .Active-passive setups (failover to backup if primary fails).Automatically switches traffic to a healthy backup resource.HC are required .
#latency based routing (Fastest AWS region_):- Route users to the region with lowest latency. route traffic to region where latency is low (latency improvement ) . for example Mumbai based traffic routes to mumbai region instead of US region.Automatically detects the AWS region with the best response time
for example :- User in Japan → routed to Tokyo region, User in Germany → routed to Frankfurt region
#gelocation based routing (Country/region based) :-Serve users based on geographic location (continent, country, or state) .Allow /restrict user to access for specific country /continent/state.Routes users from specific regions to regional endpoints.
for example: irctc site accessible only from india. for example US users → US servers,India users → India servers
#weighted routing (Split traffic by %) :- Distribute traffic across multiple resources based on weight assigned.Routes a percentage of traffic to each endpoint.
70% to server-A, 30% to server-B (for canary testing or blue/green deployments)

#VPC endpoint :-  
VPC endpoint allow to connnect to AWS services outside VPC securily (within Amazon network) .
VPC end points are created to connect to AWS services outside VPC with low cost .This ensures that traffic between your VPC and the AWS service does not leave the Amazon network, enhancing security and potentially reducing costs

#NOTE: IAM role need to be attached to ec2. 
there are 2 Types of enpoint :-
  1> gateway enpoint 2>interface endpoint  


When to Use Each
Use Gateway Endpoints:

When you need to access Amazon S3 or DynamoDB from within the same VPC and region.

For cost-effective solutions, as there are no additional charges.

When high throughput and low latency are required for S3 or DynamoDB access.

In scenarios where simple configuration and management are preferred.

Use Interface Endpoints:

When accessing AWS services other than S3 and DynamoDB.

If you require fine-grained security controls using security groups.

For cross-VPC, cross-region, or on-premises access to AWS services.

When integrating with AWS PrivateLink-enabled services.

If you need to use private DNS to access services using their standard public DNS names within your VPC.

Important config files in terraform. 
main.tf ---> Main config resource provision file.
provider.tf --> Details of provider like aws/azure. Required provider version for ex. version = "6.11.0"
variable.tf -->>variable defined here .Main.tf takes variable details from variable.tf here.
terraform.tfvars -->> values provided here as argument. Easy to edit & understand . variable.tf takes details from terraform.tfvars if not explicitlty mentioned in main.tf.
Note: Value mentioned in terraform.tfvars overwrites values in variable.tf . Values in file terraform.tfvars has more precedence over variable.tf...##Note:-if values hardcoded at main.tf has more precedence.
##Below flow 
terraform.tfvars(update values at variable.tf)------> variable.tf(updates values at main.tf)-------> main.tf 
instance_type = "t2.nano" ---> variable "instance_type" -->variable "instance_type" ---> instance_type = var.instance_type   

output.tf --> To get output for specific values. For example to get private ip id specific instance.
syntax

#main.tf syntax 
resource "aws_instance" "VM1" {
  instance_type = var.instance_type
  ami = var.ami_id
}

#variable.tf syntax
variable "ami_id" {
type = string
default = ""
}

#terraform.tfvars syntax 
ami_id  = "ami-00ca32bbc84273381"
instance_type = "t2.micro"

#output.tf syntax
output "ip" {
value = aws_instance.VM1.public_ip
}

teraform + aws behaviour 
| **Attribute**                            | **In-place Update (No Recreate)**               | **Forces Recreation (Destroy + Create New)**                  |
| ---------------------------------------- | ----------------------------------------------- | ------------------------------------------------------------- |
| `tags`                                   | ✅ Can update tags anytime                       | —                                                             |
| `user_data`                              | ✅ Updates metadata (may need reboot)            | —                                                             |
| `volume_tags`                            | ✅ In-place                                      | —                                                             |
| `disable_api_termination`                | ✅ In-place                                      | —                                                             |
| `monitoring`                             | ✅ Enable/disable detailed monitoring            | —                                                             |
| `ebs_optimized`                          | ✅ Sometimes in-place (depends on instance type) | —                                                             |
| `iam_instance_profile`                   | ✅ Attach/detach in-place                        | —                                                             |
| `credit_specification`                   | ✅ Update burstable credit option                | —                                                             |
| `ami`                                    | —                                               | ❌ Requires new instance                                       |
| `instance_type`                          | —                                               | ❌ Must recreate with new type                                 |
| `subnet_id`                              | —                                               | ❌ Moving to another subnet requires new instance              |
| `availability_zone`                      | —                                               | ❌ Must recreate in new AZ                                     |
| `key_name`                               | —                                               | ❌ New SSH key pair requires replacement                       |
| `placement_group`                        | —                                               | ❌ Changing placement group requires recreation                |
| `tenancy`                                | —                                               | ❌ Changing tenancy (default ↔ dedicated) requires recreation  |
| `host_id`                                | —                                               | ❌ Bound to dedicated host → must recreate                     |
| `root_block_device` / `ebs_block_device` | ⚠️ Some changes (tags, size) may be in-place    | ❌ Some changes (volume type, encryption) can force recreation |

############## life cycles ###################
clean ------project will be cleaned unnessary files 
validate - validate the project is correct and all necessary information is available or not
compile - compile the source code of the project (which converst .java to .class in target foolder)
test - test the compiled source code using a suitable unit testing framework. These tests should not require the code be packaged or deployed
package - take the compiled code and package it in its distributable format, such as a JAR.
install - install the package into the local repository(.m2) jar will fill generate into target folder 
deploy - done in the build environment, copies the final package to the remote repository for sharing with other developers and projects.

each process start from scratch for ex- if we start with test directly it will perform clean,validate,compile in sequence first
